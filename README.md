# PII Guardian — PII Redaction Tool

Reads a `.docx` document, detects Personally Identifiable Information (PII), replaces every detected instance with a realistic fake value, and saves the result as a new `.docx` file with all original formatting preserved.

---

## Approach

### Detection — Hybrid: Regex + spaCy NER

PII falls into two categories that require fundamentally different detection strategies:

**Structured PII** (fixed machine-readable formats) is handled by regex patterns:

| PII Type | Strategy |
|---|---|
| Email addresses | RFC-compliant regex |
| Phone numbers | Indian mobile (+91/10-digit) + international formats |
| Social Security Numbers | `\d{3}-\d{2}-\d{4}` |
| Credit card numbers | 16-digit groups |
| IP addresses | Valid IPv4 octets (0–255) |
| Dates of birth | Numeric and named-month formats |

**Linguistic PII** (context-dependent, no fixed format) is handled by spaCy's `en_core_web_lg` NER model:

| PII Type | spaCy Label |
|---|---|
| Full names | `PERSON` |
| Company names | `ORG` |
| Physical addresses | `GPE`, `LOC`, `FAC` |

The `en_core_web_lg` model was chosen over `sm`/`md` for its larger vocabulary (685k word vectors), which improves recall in dense financial document text.

### Replacement

Every detected PII value is replaced with a realistic fake generated by the [`Faker`](https://faker.readthedocs.io/) library (Indian locale `en_IN`). A global in-memory cache guarantees that the same original value is always replaced by the same fake throughout the document, preserving internal consistency.

### Formatting Preservation

`python-docx` models a paragraph as a list of `Run` objects, each with independent formatting (bold, italic, font, size). Replacement is applied at the full-paragraph text level and then redistributed back into runs, ensuring bold names and formatted fields retain their original styling.

---

## Trade-offs and Known Limitations

**False Positives (things incorrectly flagged as PII):**
- Section numbers, financial figures, and year references that match date or phone patterns
- Company abbreviations or product codes that match ORG entities in NER
- Mitigated by: sorting detections and preferring longer, more specific matches

**False Negatives (PII that may be missed):**
- Names in all-caps (e.g. table headers like `RASHI PATIL`) — NER models trained on mixed-case text perform worse on all-caps
- Highly abbreviated names (`R. Patil`) may not be recognised as PERSON
- Non-standard phone formats specific to local regions

**Formatting trade-off:**
When a single PII entity spans multiple `Run` objects (e.g. a bold name split mid-word), the replacement merges the affected text into the first run. Subsequent runs in the paragraph retain their formatting but lose their independent text segments. This is an inherent constraint of the DOCX Run model.

## Running the Tool

### Setup
```bash
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
python -m spacy download en_core_web_lg
```

### Option A: Run the Streamlit Web Application (Web Interface)
To launch the interactive dashboard, upload files visually, and download output files:
```bash
streamlit run app.py
```
This will automatically open your web browser to `http://localhost:8501`.

### Option B: Run the Command-Line Tool (CLI)
To run the processing pipeline on the default prospectus and view evaluation metrics:
```bash
python main.py --evaluate
```

## Extending to New PII Types

To add a new PII type (e.g. passport number):
1. Add a regex pattern to `PATTERNS` in `regex_detector.py`
2. Add a generator lambda to `_generate()` in `fake_generator.py`

No other files need to change.
